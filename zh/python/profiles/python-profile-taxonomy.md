# Python Profile 分类体系

本分类体系定义了实用的 Python CodeBTI Profile 系列。当用户的答案混合时，Profile 可以组合，例如 `数据优先验证者（Data-First Validator），具有测试优先集成者特征`。

## 对象中心边界守护者 (Object-Centered Boundary Keeper)

**信号：**
- 偏好对象中心的组织方式。
- 喜欢显式边界、类型化公共 API 和领域对象。
- 常选择接口、依赖注入或可替换实现。

**首选代码形态：**
- 类拥有有意义的状态和行为。
- 公共方法有类型注解并经过测试。
- 外部工具包装在适配器或门面之后。

**风险：**
- 太多小型类。
- 在变化真实出现前就定义接口。

**CodeStyle 指导：**
- 对具有行为的领域概念使用类。
- 保持对象边界显式。
- 避免只是给函数改名的类层次结构。

---

## 函数优先流水线构建者 (Function-First Pipeline Builder)

**信号：**
- 偏好小型函数和显式数据流。
- 喜欢结果对象、流水线、生成器和直接组合。
- 常选择核心逻辑测试，而非广泛的类脚手架。

**首选代码形态：**
- 纯函数或准纯函数。
- 数据显式传递。
- 流水线由命名阶段组合。

**风险：**
- 传递太多松散的字典。
- 难以调试的长链。

**CodeStyle 指导：**
- 保持函数小型，边界处有类型注解。
- 清晰地命名流水线阶段。
- 当字典变得不清晰时，使用 dataclass 或类型化记录。

---

## 数据优先验证者 (Data-First Validator)

**信号：**
- 偏好行为之前先定义数据形状。
- 喜欢 dataclass、Pydantic 风格模型、验证边界和显式载荷。
- 常选择运行时验证或类模式记录。

**首选代码形态：**
- 数据模型定义项目词汇。
- 轻量函数围绕这些模型运行。
- 无效的外部数据在边界处处理。

**风险：**
- 过度建模内部数据。
- 验证逻辑分散在各个层。

**CodeStyle 指导：**
- 对跨越模块或 IO 边界的重要数据定义模型。
- 保持验证靠近输入。
- 避免仅为复制短生命周期局部变量而创建的模型类。

---

## 务实的脚本构建者 (Pragmatic Script Builder)

**信号：**
- 偏好扁平过程式流程和低仪式感。
- 重视速度、可读辅助函数和最小结构。
- 常选择简单依赖管理和轻量测试。

**首选代码形态：**
- 最初只有少量文件。
- 直接函数和模块常量。
- 仅在出现重复或混淆时才重构。

**风险：**
- 隐藏的共享状态。
- 如果项目快速增长，模块难以扩展。

**CodeStyle 指导：**
- 简单起步，但设置清晰的提取阈值。
- 除非会话记录明确允许，否则避免全局变更。
- 在重大重构前，为核心行为添加测试。

---

## 测试优先集成者 (Test-First Integrator)

**信号：**
- 偏好广泛测试覆盖、集成测试、严格边界或 CI 友好工作流。
- 常选择依赖注入、适配器和常规 Git 实践。

**首选代码形态：**
- 依赖在测试中可替换。
- 外部系统位于清晰边界之后。
- 新行为包含在实现之前或同时的测试。

**风险：**
- 过多测试脚手架导致迭代变慢。
- Mock 密集型测试不能保护真实行为。

**CodeStyle 指导：**
- 要求对公共行为和重要失败路径进行测试。
- 偏好稳定边界的集成测试。
- 测试替身靠近其替换的适配器或接口。

---

## 框架对齐构建者 (Framework-Aligned Builder)

**信号：**
- 偏好框架或工具约定用于文件夹布局、依赖管理和项目结构。
- 重视生态兼容性，而非自定义架构。

**首选代码形态：**
- 遵循所选框架期望的布局。
- 除非框架边界不足，否则保持本地抽象薄。
- 使用框架原生配置和测试模式。

**风险：**
- 用自定义模式对抗框架。
- 无明确原因的锁定。

**CodeStyle 指导：**
- 首先遵循框架约定。
- 仅在外部服务或项目特定规则需要隔离时添加适配器。
- 记录与框架的偏差。

---

## 算法优先极简主义者 (Algorithm-First Minimalist)

**信号：**
- 偏好性能、直接 API、数据结构和低依赖数。
- 常选择算法优先架构和围绕核心逻辑的测试。

**首选代码形态：**
- 函数靠近它们操作的数据结构。
- 仅在移除真实复杂度时才添加抽象。
- 避免依赖，除非它们明确改善正确性或速度。

**风险：**
- 非显而易见的算法文档不足。
- 忽略集成和面向用户的失败模式。

**CodeStyle 指导：**
- 保持热路径直接和可测量。
- 算法选择不明显时加以解释。
- 为边界情况和复杂度敏感行为添加针对性测试。
