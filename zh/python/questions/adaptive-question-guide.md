# 适应性追问指南

在 10 个固定 Python 问题之后，Agent 必须提出恰好 5 个适应性后续追问。

适应性追问应澄清用户最强的偏好、矛盾或项目特定风险。它们应使用与 [question-format.md](question-format.md) 相同的题目卡结构，但场景和代码示例可根据用户项目描述和固定答案生成。

## 必做行为

- 提出恰好 5 个适应性问题。
- 保持问题基于示例。
- 当代码使权衡更容易判断时，使用 Python 代码片段。
- 对用户隐藏评分。
- 首先将答案映射到风格维度，其次映射到模式信号。
- 不问通用的人格问题。
- 不让用户选择已命名的设计模式。

## 追问触发器

根据以下触发器生成追问：

- 模糊：用户的固定答案在相互竞争的风格之间过于平均分配。
- 矛盾：用户选择的偏好可能在实现中冲突。
- 项目风险：项目描述包含 IO、并发、外部 API、安全、数据验证或长期扩展。
- 强烈信号：多个答案指向相同的风格轴或模式族。
- 模式误用风险：用户似乎倾向于使用简单 Python 可能更清晰的重量级抽象。

## 潜在的追问方向

在审阅项目描述和 10 个固定答案后，选择恰好 5 个方向。优先选择能改变生成的 `CodeStyle.md` 的方向。

### 方向一：澄清默认形态

当 Q1 或后续答案未清楚表明用户想要类中心、函数优先、数据优先还是过程式代码时使用。

询问一个小型真实工作流，并比较首选的项目形态：

- 带方法的对象，
- 基于显式数据的一组函数，
- 数据模型加小型服务函数，
- 扁平脚本式模块。

### 方向二：外部工具边界

当项目涉及 API、数据库、文件、模型提供者、CLI 工具、Web 框架或第三方库时使用。

询问用户是希望直接调用依赖、小型适配器，还是围绕子系统的稳定门面。

### 方向三：增长与扩展

当项目可能添加新的后端、格式、算法、规则类型、命令或插件时使用。

询问用户是倾向于在增长真实出现前保持简单分支、命名策略函数/类，还是显式的注册表/插件形态。

### 方向四：数据信任与验证

当项目处理用户输入、外部文件、API 响应、研究数据或配置时使用。

询问无效数据应在何处处理：

- 靠近来源，
- 在领域边界，
- 在每个需要安全的函数内部。

### 方向五：错误与恢复风格

当故障可能是可恢复的或面向用户时使用。

询问代码是应通过异常快速失败、返回显式结果对象，还是在运行主工作流前验证。

### 方向六：状态与历史

当项目包含会话、多步工作流、检查点、撤销、缓存或长运行时进程时使用。

询问状态应保持在本地可变状态、显式状态对象，还是作为快照/历史捕获。

### 方向七：测试与审查严格度

当答案显示快速推进与维护稳定行为之间存在张力时使用。

询问哪种测试应阻止未来的 Agent 编辑：

- 主流程的冒烟测试，
- 用户可见结果的业务测试，
- 重要边界的契约测试。

### 方向八：仪式预算

当用户选择了多个抽象密集型选项时使用。

询问他们希望 Agent 在何处停止添加结构：

- 停止在小型辅助函数，
- 仅在真实边界允许接口，
- 当未来变化可能时允许类模式结构。

### 方向九：可调试性

当用户喜欢流水线、事件、回调、注册表或插件时使用。

询问他们是喜欢直接的逐步流程、命名的流水线阶段，还是解耦的事件式流程。

### 方向十：复用与本地清晰度

当项目可能在模块之间或未来项目中共享代码时使用。

询问重复代码是应保留在本地直到造成痛苦、尽早成为共享辅助函数，还是成为可复用的框架式组件。

## 需要澄清的常见矛盾

- 严格类型偏好加上高度动态的插件行为。
- 全局便利性加上强测试隔离。
- 最小抽象加上许多计划的实现。
- 快速失败异常加上面向用户的可恢复工作流。
- 事件驱动解耦加上对逐步调试的期望。
- 分阶段构建器加上对简短直接代码的偏好。

## 适应性问题的格式

每个适应性问题应包含：

- `Dimension`（维度）
- `User-facing scenario`（用户面对的场景）
- `Code example`（代码示例）
- `Choices`（选项）
- `Agent scoring`（代理评分）
- `Pattern signals`（模式信号）
- `CodeStyle output implications`（CodeStyle 输出影响）

只有当代码使问题更不清楚时，Agent 才可省略代码示例。

## 适应性追问后的输出

在所有 15 个问题都回答后，Agent 应总结：

- 主要风格维度，
- 较弱或矛盾的维度，
- 鼓励的模式，
- 谨慎使用的模式，
- 应避免的模式，
- `CodeStyle.md` 的初步起草方向。

此总结应在生成最终项目指导之前记录在会话记录中。
