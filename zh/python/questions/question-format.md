# 题目卡格式

将此格式用于固定和适应性 CodeBTI 问题。

````md
## QN. 占位符标题

Dimension:
此问题探测的主要 CodeBTI 维度。

User-facing scenario:
向用户展示的一两句话场景。

User-facing instruction:
选择您最自然愿意维护的风格。您可以随时修改之前的答案。

Code example:
```python
# 可选。保持简短且聚焦。
```

Choices:
- A. 一个实际可维护的选项。
- B. 一个不同的实际可维护的选项。
- C. 另一个实际可维护的选项。

Agent scoring:
- A: 隐藏的风格轴解释。
- B: 隐藏的风格轴解释。
- C: 隐藏的风格轴解释。

Pattern signals:
- A: 可选的模式倾向或反模式信号。
- B: 可选的模式倾向或反模式信号。
- C: 可选的模式倾向或反模式信号。

CodeStyle output implications:
答案应如何影响生成的项目指导。
````

## 代码示例规则

- 初始版本使用 Python。
- 保持片段足够小以快速比较。
- 尽可能用不同风格展示同一问题。
- 优先真实项目场景：配置、数据加载、验证、服务、命令、插件、测试和错误处理。
- 除非问题关于依赖容忍度，否则避免需要外部库的示例。
- 不要让一个选项明显错误。
- 不要让用户识别已命名的设计模式。

## 选项规则

- 使用 2-4 个选项。
- 使每个选项对某些项目都可信。
- 描述用户愿意维护什么，而非学术上"正确"的是什么。
- 保持标签中立且具体。
- 避免在用户面对的选项中使用仅术语标签，如"Strategy"或"Factory"。
- 避免暗示技能水平的标签，如"高级"、"严格"、"简单"或"快速粗暴"。
- 除非用户询问 CodeBTI 工作原理，否则保持评分隐藏。

## 记录规则

- 提问前，将完整用户面对的卡片保存到 `Recording.md`。
- 记录标题、维度、场景、说明、代码示例（如果有）、选项和来源路径。
- 对于适应性问题，记录生成的题目文本，精确度足以让另一个 Agent 在没有聊天历史的情况下理解用户回答了什么。
- 回答后，记录所选选项、用户笔记、给出的反馈和隐藏推理笔记。

## 评分规则

在模式名称之前，按风格维度评分答案。有用的维度包括：

- 抽象深度，
- 类型姿态，
- 数据建模偏好，
- 构造风格，
- 依赖边界严格度，
- 错误处理风格，
- 扩展机制，
- 工作流编排，
- 状态管理，
- 测试和文档期望。

模式信号应保持为次要证据。生成的 `CodeStyle.md` 应说明代码中该怎么做，而非仅命名模式。
