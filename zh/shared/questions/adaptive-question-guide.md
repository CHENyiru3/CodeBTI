# 适应性追问指南

在固定的项目问题和语言问题之后，Agent 必须在整个 CodeBTI 会话中提出恰好 5 个适应性后续追问。

适应性问题应澄清用户最强的偏好、矛盾或项目特定风险。它们应使用与 [question-format.md](question-format.md) 相同的题目卡结构，但场景和代码示例可以根据项目描述、项目级答案和语言级答案生成。

## 必做行为

- 整个会话总共提出恰好 5 个适应性问题。
- 保持问题基于示例。
- 当代码能让权衡更容易判断时，使用目标语言的代码片段。
- 当问题涉及协作、验证、输出、依赖或多语言边界时，使用项目级场景。
- 对用户隐藏评分。
- 首先将答案映射到风格维度，其次映射到模式信号。
- 不问通用人格问题。
- 不让用户选择已命名的设计模式。

## 追问范围

每个适应性问题应有一个范围：

- `Project`：跨项目的工作流、验证、依赖、输出或记录决策。
- `Language:<name>`：针对某个语言包的问题，例如 Python 或 TypeScript。
- `Cross-language`：关于两个或多个语言部分如何共享或覆盖规则的问题。

在 `Recording.md` 中记录该范围。

## 追问触发器

根据以下触发器生成追问：

- 模糊：固定答案在相互竞争的风格之间过于平均。
- 矛盾：所选偏好在实现中可能冲突。
- 项目风险：项目包含 IO、并发、外部 API、安全、数据验证、生成内容或长期扩展。
- 强信号：多个答案指向同一风格轴或模式族。
- 共享边界风险：语言特定偏好与项目级默认规则冲突。
- 模式误用风险：用户倾向于重抽象，而直接的语言习惯用法可能更清晰。

## 潜在追问方向

在审阅项目描述、项目级答案和语言级答案后，选择恰好 5 个方向。优先选择会改变生成指导的方向。

### 方向一：澄清默认代码形态

当某个语言轮次没有清楚显示用户想要类中心、函数优先、数据优先、接口优先还是过程式代码时使用。

询问一个小型真实工作流，并比较首选形态：

- 带方法的领域对象，
- 基于显式数据的一组函数，
- 数据模型加小型服务函数，
- 扁平脚本式模块，
- 带实现的接口或协议边界。

### 方向二：外部工具边界

当项目提到 API、数据库、文件、模型提供方、CLI 工具、Web 框架或第三方库时使用。

询问用户想要直接调用依赖、小型适配器，还是围绕子系统建立稳定门面。

### 方向三：增长与扩展

当项目可能增加新后端、格式、算法、规则类型、命令或插件时使用。

询问用户偏好在增长真实出现前保持简单分支、命名策略函数/类，还是显式注册表/插件形态。

### 方向四：数据可信度与验证

当项目处理用户输入、外部文件、API 响应、研究数据、生成内容或配置时使用。

询问无效数据应在哪里处理：

- 靠近来源，
- 在领域边界，
- 在每个需要安全性的函数内部。

### 方向五：错误与恢复风格

当失败可能可恢复或面向用户时使用。

询问代码应通过异常快速失败、返回显式结果对象，还是在运行主流程前验证。

### 方向六：状态与历史

当项目包含会话、多步工作流、检查点、撤销、缓存或长时间运行流程时使用。

询问状态应保持局部可变、放入显式状态对象，还是作为快照/历史捕获。

### 方向七：测试与审查严格度

当答案显示快速推进与稳定行为之间存在张力时使用。

询问什么测试或检查应阻止未来 Agent 编辑：

- 主流程冒烟测试，
- 用户可见行为测试，
- 重要边界的契约测试，
- 文档结构验证。

### 方向八：仪式成本

当用户选择多个偏重抽象的选项时使用。

询问 Agent 应在哪里停止增加结构：

- 停在小型辅助函数，
- 只在真实边界允许接口，
- 仅在未来变化可能真实存在时允许模式化结构。

### 方向九：可调试性

当用户喜欢管线、事件、回调、注册表或插件时使用。

询问用户偏好直接逐步流程、命名管线阶段，还是解耦事件式流程。

### 方向十：复用与局部清晰度

当项目可能跨模块、语言或未来项目共享代码时使用。

询问重复代码应保留在本地直到痛点明显、较早变成共享辅助函数，还是变成可复用框架式组件。

## 常见矛盾

- 严格类型偏好与高度动态插件行为。
- 全局便利性与强测试隔离。
- 最小抽象与许多计划中的实现。
- 快速失败异常与面向用户的可恢复流程。
- 事件驱动解耦与逐步调试需求。
- 分阶段构建器与短直接代码偏好。
- 共享项目默认规则与语言特定框架约定。
- 轻量流程与 CI 必须通过或发布管理预期。

## 适应性题目格式

每个适应性问题应包含：

- `Dimension`
- `Scope`
- `User-facing scenario`
- `User-facing instruction`
- `Code example`
- `Choices`
- `Agent scoring`
- `Pattern signals`
- `CodeStyle output implications`

当代码会降低清晰度时，Agent 可以省略代码示例。

## 适应性问题后的输出

记录全部适应性答案后，Agent 应总结：

- 主要项目级工作流信号，
- 主要语言特定风格维度，
- 较弱或冲突的维度，
- 鼓励的模式，
- 谨慎允许的模式，
- 默认避免的模式，
- 以及生成指导的第一版方向。

该总结应在生成最终项目指导前记录到会话记录中。
