# 题目卡格式

将此格式用于固定和适应性 CodeBTI 问题。

````md
## QN. 占位符标题

Question ID:
稳定标识符，例如 `project.validation.gate`、`python.error.policy` 或 `typescript.type.boundary`。

Scope:
Project、Language:<name> 或 Cross-language。

Dimension:
此问题探测的主要 CodeBTI 维度。

User-facing scenario:
向用户展示的一两句场景。

User-facing instruction:
选择您最自然愿意维护的风格。您可以随时修改之前的答案。

Code example:
```text
可选。当代码能让权衡更容易判断时，使用目标语言。
```

Choices:
- A. 一个实际可维护的选项。
- B. 一个不同的实际可维护的选项。
- C. 另一个实际可维护的选项。

Agent scoring:
- A: 隐藏的风格轴解释。
- B: 隐藏的风格轴解释。
- C: 隐藏的风格轴解释。

Pattern signals:
- A: 可选的模式倾向或反模式信号。
- B: 可选的模式倾向或反模式信号。
- C: 可选的模式倾向或反模式信号。

CodeStyle output implications:
答案应如何影响生成的项目指导。
````

## 代码示例规则

- 语言特定问题使用目标语言。
- 项目级流程问题使用文本示例或文件树示例。
- 保持片段足够短，便于快速比较。
- 尽可能展示同一问题的不同风格解法。
- 优先使用真实项目情境：配置、数据加载、验证、服务、命令、插件、测试、工作流、输出、依赖和错误处理。
- 除非问题讨论依赖容忍度，否则避免需要外部库的示例。
- 不要让某个选项显然错误。
- 不要要求用户识别已命名的设计模式。

## 选项规则

- 使用 2-5 个选项。
- 每个选项都应适用于某些真实项目。
- 描述用户会维护什么，而不是学术上什么是“正确”的。
- 标签保持中性且具体。
- 避免只靠术语的标签，例如 "Strategy" 或 "Factory"。
- 避免暗示能力等级的标签，例如“高级”“严格”“简单”“快速粗糙”。
- 除非用户要求查看 CodeBTI 如何工作，否则隐藏评分。

## 记录规则

- 提问前，将完整用户面对的题目卡保存到 `Recording.md`。
- 记录标题、题目 ID（如有）、范围、维度、场景、说明、代码示例（如有）、选项和来源路径。
- 对适应性问题，记录生成的问题文本，使另一个 Agent 不依赖聊天历史也能理解用户回答了什么。
- 回答后，记录所选选项、用户备注、给出的反馈和隐藏推理笔记。

## 评分规则

先按风格维度评分，再看模式名称。常用维度包括：

- 项目控制模型，
- 输出形态，
- 验证门禁，
- 共享规则与语言规则边界，
- 依赖治理，
- 变更记录政策，
- 抽象深度，
- 类型姿态，
- 数据建模偏好，
- 构造风格，
- 依赖边界严格度，
- 错误处理风格，
- 扩展机制，
- 工作流编排，
- 状态管理，
- 测试和文档预期。

模式信号应始终是辅助证据。生成的 `CodeStyle.md` 应说明代码中该怎么做，而不只是命名模式。
